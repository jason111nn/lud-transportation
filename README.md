# 🚦 低頭顯示器 - 交通號誌提醒系統 (LUD System)
> **LUD (Low-head Up Display)** 是我為這套專案設計的縮寫，核心概念是將駕駛員的抬頭顯示器（HUD）邏輯，轉化為守護低頭族安全的技術。

這是一套基於 **IoT 物聯網技術** 的智慧交通解決方案，解決「Smombie (低頭族)」在路口等紅燈滑手機，導致綠燈亮起時起步延遲或後車追撞的痛點。我們不強迫你抬頭，而是把號誌「主動」送到你手機裡。

---

## ⚡ 核心技術：快、輕、準

* **V2I (Vehicle-to-Infrastructure) 協作**：號誌端（RSU）主動推播訊息，實現基礎設施與個人設備的直接溝通。
* **極低延遲藍牙廣播 (BLE Advertising)**：採用 **Non-connectable** 廣播模式，實測延遲低於 **$200\text{ms}$**，確保手機端的秒數與路口燈號體感零時差同步。
* **自定義 5-Byte 極簡協定**：將識別碼、相位與秒數封裝在極小封包中，提升在高干擾環境下的傳輸穩定性。
* **Vue 3 + GSAP 監控儀表板**：前端採用現代化開發架構，將硬體端的 RAW 封包數據轉化為高質感的動態監控介面。

---

## 📊 數據通訊協定 (5-Byte Protocol)

為了達成輕量化傳輸，我們將所有數據壓縮在 5 個 Byte 內：

| 位元組 | 功能 | 說明 |
| :--- | :--- | :--- |
| **Byte 0-1** | **識別碼** | 固定傳送 `0xFFFF`，確保 App 只接收來自 LUD 系統的數據。 |
| **Byte 2-3** | **號誌相位** | 傳輸路口 A 與 B 兩側的即時燈號狀態。 |
| **Byte 4** | **即時倒數** | 載入目前的剩餘秒數，供接收端即時顯示。 |

## 🛠️ 實作挑戰與解決方案 (Q&A)

針對實際部署可能遇到的挑戰，我們在設計階段就預留了技術對策：

* **硬體如何整合？**
    不依賴影像辨識，而是直接從交通控制箱擷取信號電平。透過 **光耦合器 (Optocoupler)** 進行硬體隔離，安全地讀取燈號狀態。
* **多向路口處理？**
    5-Byte 協定具備高度擴充性，利用 **位元遮罩 (Bitmask)** 技術，Byte 2-3 可承載多相位狀態，配合 App 端地理圖資即可解析複雜路口。
* **秒數異動同步？**
    系統採用「主動推播」機制。不論路口紅綠燈週期如何變動，ESP32 每一秒都會擷取最新數據並廣播，App 永遠顯示最新數據，無需預載路口參數。
